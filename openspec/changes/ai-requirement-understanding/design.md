## 背景

平台当前在 `raw_content` 字段中存储需求文档，没有结构化解析或 AI 分析。需求以多种格式到达（带图片的 PDF、Markdown、截图、URL、GitLab issue、AI 对话），使得下游 AI 代理难以一致地消费它们。

现有基础设施：
- 带有 `raw_content` 和 `ai_understanding_result` 字段的 `AiUnderstanding` 模型
- 处理需求的 Celery 任务 `generate_ai_understanding()`
- integrations 中的 `AIService`，支持 DeepSeek/Claude 降级
- 用于组织需求的模块系统

当前痛点：
- PDF 中的图片未被提取或识别
- AI 分析中未包含历史上下文（模块模式、相关需求）
- AI 理解前没有质量控制关卡
- 下游代理（技术设计、测试生成）需要重新解析文档

## 目标 / 非目标

**目标：**
- 在所有输入格式中实现一致的文档解析
- 为下游 AI 消费提供结构化、丰富的需求理解
- 在关键质量关卡（解析、理解）支持人工审核
- 使图片识别策略可配置以进行 A/B 测试
- 随时间积累模块知识以获得更好的上下文
- 自动检测质量问题（缺失信息、矛盾、风险）

**非目标：**
- 需求的实时协作编辑
- 需求文档的版本控制（改用 GitLab 集成）
- 无人工审核的自动需求批准
- 支持视频或音频需求输入

## 决策

### 1. 两阶段处理：解析 → 理解

**决策**：拆分为独立阶段，中间有人工审核。

**理由**：
- 解析质量直接影响 AI 理解质量
- 允许人工在昂贵的 AI 处理前验证图片识别结果
- 支持重新解析而无需重新运行 AI 理解
- 清晰的关注点分离：提取 vs. 分析

**考虑的替代方案**：
- 单阶段：一步完成解析和理解 → 拒绝，因为解析错误会在没有可见性的情况下污染 AI 输出
- 三阶段（解析 → 提取 → 理解）→ 拒绝，过度工程化

### 2. 解析期间的图片识别（可配置）

**决策**：在解析阶段识别图片，存储在单独字段，使其可配置。

**理由**：
- 图片包含关键信息（带标注的 UI 原型、流程图）
- 人工可以在 AI 理解前审核识别质量
- 配置允许比较"纯文本"与"文本+图片"方法
- 批量处理（最多 15 张图片）优化 Vision LLM 成本

**考虑的替代方案**：
- 在 AI 理解期间识别 → 拒绝，因为无法人工审核识别质量
- 始终跳过图片 → 拒绝，因为会丢失有价值的信息
- 提取图片但不识别 → 作为配置选项考虑（已实现）

**实现**：
```python
AI_UNDERSTANDING_CONFIG = {
    'enable_image_recognition': True,  # 切换图片识别
    'use_content_field': 'parsed_content_with_images',  # AI 使用哪个字段
    'process_images_in_understanding': False  # 在理解阶段重新处理
}
```

### 3. 双内容字段

**决策**：同时存储 `parsed_content`（文本+表格）和 `parsed_content_with_images`（文本+表格+图片识别）。

**理由**：
- 支持 A/B 测试：比较有/无图片的 AI 理解质量
- 为成本敏感场景保留纯文本版本
- 允许在不重新解析的情况下切换策略

**考虑的替代方案**：
- 带图片的单字段 → 拒绝，因为无法比较方法
- 三个字段（文本、表格、图片分开）→ 拒绝，过于复杂

### 4. 历史上下文丰富

**决策**：向 AI 理解注入两层上下文：
1. 模块知识（从过去需求积累的模式）
2. 相关需求（AI 推荐的连接）

**理由**：
- 模块知识捕获约定（例如，"auth 模块始终使用 JWT"）
- 相关需求确保一致性（例如，"导出应匹配导入格式"）
- 上下文显著提高 AI 理解质量
- 增量构建知识库

**实现**：
- `ModuleKnowledge` 模型存储每个模块的积累见解
- `RequirementRelation` 模型存储 AI 推荐的关系
- 两者都注入 AI 理解提示

### 5. 质量问题检测

**决策**：AI 在理解阶段检测并结构化质量问题。

**理由**：
- 早期检测防止下游问题
- 结构化格式（类型、描述、严重性）支持自动处理
- 人工审核员在批准前看到问题

**问题类型**：
- 缺失：不完整的信息
- 矛盾：冲突的陈述
- 未定义：模糊的术语
- 风险：潜在问题

### 6. Given-When-Then 验收标准

**决策**：以 Given-When-Then 格式构建验收标准。

**理由**：
- 开发人员和测试人员熟悉的标准 BDD 格式
- 直接转换为测试用例
- 强制需求规范的清晰性

### 7. 重要性级别（核心/重要/辅助）

**决策**：在 MoSCoW 优先级之外添加重要性分类。

**理由**：
- MoSCoW 是关于交付优先级
- 重要性是关于系统关键性
- 帮助下游代理（例如，测试生成专注于核心/重要）

### 8. Celery 异步处理

**决策**：对解析和 AI 理解任务都使用 Celery。

**理由**：
- 解析大型 PDF 可能需要 10-30 秒
- 图片的 Vision LLM 调用每批增加 5-15 秒
- 带上下文的 AI 理解可能需要 20-60 秒
- 异步防止 API 超时，支持进度跟踪

## 风险 / 权衡

**[风险]** Vision LLM 图片识别对于复杂图表可能不准确
→ **缓解**：解析后的人工审核关卡；支持重新解析；配置允许禁用

**[风险]** 批量处理 15 张以上图片增加延迟
→ **缓解**：异步处理；进度指示器；可配置批量大小

**[风险]** 模块知识可能随时间变得过时
→ **缓解**：时间戳知识条目；定期审核机制；允许手动更新

**[风险]** AI 可能错误识别相关需求
→ **缓解**：存储置信度分数；人工可以在 UI 中删除错误链接；不阻塞工作流

**[权衡]** 双字段增加存储成本
→ **接受**：存储便宜；灵活性值得；可以稍后清理旧字段

**[权衡]** 两阶段处理增加总时间
→ **接受**：质量比速度更重要；人工审核防止昂贵的返工

**[权衡]** Vision LLM 调用增加 API 成本
→ **缓解**：可配置（可以禁用）；批量处理优化；成本跟踪

## 迁移计划

**阶段 1：数据库模式**
1. 添加新字段到 `AiUnderstanding`（所有可空）
2. 扩展 `Module` 模型
3. 创建 `ModuleKnowledge` 和 `RequirementRelation` 模型
4. 运行迁移

**阶段 2：后端任务**
1. 实现 `parse_document_task()` 及所有格式解析器
2. 修改 `generate_ai_understanding()` 使用解析内容
3. 添加模块知识和相关需求获取
4. 在 `ai_prompts.py` 中创建提示模板

**阶段 3：API 端点**
1. 添加解析审核端点
2. 添加重新解析触发端点
3. 添加理解批准端点
4. 添加下游消费端点

**阶段 4：前端集成**
1. 更新需求详情页面，添加两标签布局（解析内容 / AI 理解）
2. 添加解析结果的审核 UI
3. 添加 AI 理解的批准 UI
4. 突出显示质量问题

**阶段 5：配置和测试**
1. 将 `AI_UNDERSTANDING_CONFIG` 添加到 settings
2. 测试所有 4 种配置组合
3. 比较质量指标
4. 记录推荐设置

**回滚策略**：
- 新字段可空；旧代码继续工作
- 可以通过配置禁用功能
- 解析阶段是可选的；可以直接跳到 AI 理解
- 对现有 API 没有破坏性更改

## 未决问题

1. 是否应该支持增量解析（PDF 更新时仅解析新页面）？
   - **倾向于**：MVP 不支持；如果用户请求则添加

2. 模块知识应该版本化还是仅时间戳？
   - **倾向于**：MVP 使用时间戳；版本化增加复杂性

3. 是否应该在批准后自动触发下游代理（技术设计、测试生成）？
   - **倾向于**：MVP 手动触发；稍后通过配置自动触发

4. 是否应该支持每个项目/模块的自定义提示模板？
   - **倾向于**：MVP 不支持；单一模板更易维护
