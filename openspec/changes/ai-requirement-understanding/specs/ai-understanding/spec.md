## 新增需求

### 需求：生成结构化功能列表
系统必须分析解析后的需求内容并生成结构化的功能列表。

#### 场景：从需求文本中提取功能
- **当** AI 理解处理解析后的内容
- **则** 系统生成包含名称、描述和重要性级别的功能列表

#### 场景：分类功能重要性
- **当** 生成功能列表
- **则** 系统为每个功能分配重要性级别（核心/重要/辅助）

#### 场景：分配 MoSCoW 优先级
- **当** 生成功能列表
- **则** 系统为每个功能分配 MoSCoW 优先级（Must/Should/Could/Won't）

### 需求：生成 Given-When-Then 验收标准
系统必须为每个功能生成 Given-When-Then 格式的验收标准。

#### 场景：为功能生成验收标准
- **当** 处理一个功能
- **则** 系统生成至少一个 Given-When-Then 结构的验收标准

#### 场景：处理没有明确验收标准的功能
- **当** 功能缺乏可测试条件
- **则** 系统标记质量问题并生成尽力而为的验收标准

### 需求：识别需求变更
系统必须分析相比现有功能修改、新增或删除了什么。

#### 场景：识别修改的功能
- **当** 需求更新现有功能
- **则** 系统列出修改的功能及变更描述

#### 场景：识别新增功能
- **当** 需求引入新功能
- **则** 系统列出新增的功能

#### 场景：识别删除的功能
- **当** 需求废弃功能
- **则** 系统列出删除的功能及原因

### 需求：检测质量问题
系统必须自动检测需求内容中的质量问题。

#### 场景：检测缺失信息
- **当** 需求缺少关键细节
- **则** 系统标记类型为'缺失'的问题并描述缺少什么

#### 场景：检测矛盾
- **当** 需求包含冲突的陈述
- **则** 系统标记类型为'矛盾'的问题并描述矛盾内容

#### 场景：检测未定义术语
- **当** 需求使用模糊或未定义的术语
- **则** 系统标记类型为'未定义'的问题并列出未定义术语

#### 场景：检测风险
- **当** 需求存在潜在实现风险
- **则** 系统标记类型为'风险'的问题并描述风险

### 需求：使用模块知识丰富上下文
系统必须将积累的模块知识注入 AI 理解上下文。

#### 场景：包含模块模式
- **当** 需求属于有积累知识的模块
- **则** 系统在 AI 提示中包含模块模式和约定

#### 场景：处理没有知识的模块
- **当** 需求属于没有积累知识的新模块
- **则** 系统在没有模块上下文的情况下继续

### 需求：使用相关需求丰富上下文
系统必须识别并包含相关需求作为上下文。

#### 场景：按模块查找相关需求
- **当** 生成 AI 理解
- **则** 系统包含同一模块的最近需求

#### 场景：按 AI 推荐查找相关需求
- **当** AI 识别相关需求
- **则** 系统在 RequirementRelation 模型中存储关系

### 需求：使用可配置的内容源
系统必须使用配置的内容字段（包含或不包含图片）进行 AI 理解。

#### 场景：使用纯文本内容
- **当** 配置将 use_content_field 设置为 'parsed_content'
- **则** 系统使用 parsed_content 字段进行 AI 理解

#### 场景：使用包含图片识别的内容
- **当** 配置将 use_content_field 设置为 'parsed_content_with_images'
- **则** 系统使用 parsed_content_with_images 字段进行 AI 理解

### 需求：生成结构化 JSON 输出
系统必须以结构化 JSON 格式输出 AI 理解。

#### 场景：生成完整的理解结构
- **当** AI 理解完成
- **则** 系统存储包含 features、changes、acceptance_criteria 和 quality_issues 的 JSON

#### 场景：在输出中包含元数据
- **当** 生成理解
- **则** 系统在输出中包含模块上下文和相关需求

### 需求：支持思维链推理
系统必须使用思维链提示以获得更好的分析质量。

#### 场景：在输出前生成推理
- **当** AI 处理需求
- **则** 系统在结构化输出前包含思考部分

### 需求：跟踪理解状态
系统必须使用状态字段跟踪 AI 理解进度。

#### 场景：初始化理解任务
- **当** AI 理解开始
- **则** 系统将 ai_status 设置为 'processing'

#### 场景：成功完成理解
- **当** 理解无错误完成
- **则** 系统将 ai_status 设置为 'done' 并存储结果

#### 场景：处理理解失败
- **当** 理解遇到错误
- **则** 系统将 ai_status 设置为 'failed' 并存储错误信息

### 需求：启用 AI 理解的人工审核
系统必须允许人工审核员在批准前验证 AI 理解质量。

#### 场景：提交理解供审核
- **当** AI 理解成功完成
- **则** 系统将 ai_reviewed 标记为 False 并通知审核员

#### 场景：批准 AI 理解
- **当** 审核员批准理解
- **则** 系统将 ai_reviewed 设置为 True 并记录审核员和时间戳

#### 场景：触发重新生成
- **当** 审核员拒绝理解并请求重新生成
- **则** 系统将 ai_status 重置为 'pending' 并触发新的理解任务

### 需求：积累模块知识
系统必须从批准的 AI 理解中提取并存储可重用的知识。

#### 场景：提取模块模式
- **当** AI 理解被批准
- **则** 系统提取模式并存储在 ModuleKnowledge 模型中

#### 场景：更新现有模块知识
- **当** 新理解提供更新的模式
- **则** 系统更新现有的 ModuleKnowledge 条目
