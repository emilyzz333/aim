## 新增需求

### 需求：解析 PDF 文档
系统必须能够从 PDF 文档中提取文本、表格和图片。

#### 场景：解析包含文本和表格的 PDF
- **当** 用户上传包含文本和表格的 PDF
- **则** 系统提取所有文本内容和表格结构到 parsed_content 字段

#### 场景：解析包含嵌入图片的 PDF
- **当** 用户上传包含图片的 PDF
- **则** 系统提取图片并单独存储，包含位置元数据

#### 场景：解析包含混合内容的 PDF
- **当** 用户上传包含文本、表格和图片的 PDF
- **则** 系统提取所有内容类型并保留其相对顺序

### 需求：解析 DOCX 文档
系统必须能够从 Microsoft Word 文档中提取文本、表格和图片。

#### 场景：解析包含格式化文本的 DOCX
- **当** 用户上传包含标题、段落和列表的 DOCX
- **则** 系统提取文本并保留结构（标题、列表）

#### 场景：解析包含表格的 DOCX
- **当** 用户上传包含表格的 DOCX
- **则** 系统以结构化格式提取表格内容

### 需求：解析 Markdown 文档
系统必须能够处理 Markdown 文档并提取结构化内容。

#### 场景：解析包含图片的 Markdown
- **当** 用户上传包含图片引用的 Markdown
- **则** 系统提取文本并下载引用的图片

#### 场景：解析包含表格的 Markdown
- **当** 用户上传包含表格语法的 Markdown
- **则** 系统将表格解析为结构化格式

### 需求：解析 URL 内容
系统必须能够从提供的 URL 获取并解析内容。

#### 场景：解析网页内容
- **当** 用户提供网页 URL
- **则** 系统获取 HTML，转换为 Markdown，并提取图片

#### 场景：处理 URL 获取失败
- **当** URL 无法访问或返回错误
- **则** 系统记录解析失败并保存错误信息

### 需求：解析 GitLab issue
系统必须能够通过 API 获取并解析 GitLab issue 内容。

#### 场景：解析包含描述的 GitLab issue
- **当** 用户提供 GitLab issue URL
- **则** 系统通过 API 获取 issue 并提取标题、描述和评论

#### 场景：解析包含附件的 GitLab issue
- **当** GitLab issue 包含图片附件
- **则** 系统下载附件并包含在解析内容中

### 需求：解析 AI 对话内容
系统必须能够处理 AI 对话记录作为需求输入。

#### 场景：解析包含用户和 AI 消息的对话
- **当** 用户提供 AI 对话记录
- **则** 系统从对话上下文中提取需求详情

### 需求：使用 Vision LLM 识别图片
系统必须能够在配置启用时使用 Vision LLM 识别图片内容。

#### 场景：识别 UI 原型图片
- **当** 图片识别已启用且文档包含 UI 原型
- **则** 系统将图片发送到 Vision LLM 并存储文本描述

#### 场景：批量处理多张图片
- **当** 文档包含超过 15 张图片
- **则** 系统以 15 张为一批处理图片以遵守 Vision LLM 限制

#### 场景：禁用时跳过图片识别
- **当** 配置中禁用图片识别
- **则** 系统提取图片但不发送到 Vision LLM

### 需求：存储双内容字段
系统必须同时存储纯文本和包含图片识别的解析内容。

#### 场景：存储不含图片的解析内容
- **当** 解析完成
- **则** 系统将文本和表格存储在 parsed_content 字段

#### 场景：存储包含图片识别的解析内容
- **当** 解析完成且图片识别已启用
- **则** 系统将文本、表格和图片描述存储在 parsed_content_with_images 字段

### 需求：跟踪解析状态
系统必须使用状态字段跟踪解析进度。

#### 场景：初始化解析任务
- **当** 文档解析开始
- **则** 系统将 parse_status 设置为 'processing'

#### 场景：成功完成解析
- **当** 解析无错误完成
- **则** 系统将 parse_status 设置为 'done'

#### 场景：处理解析失败
- **当** 解析遇到错误
- **则** 系统将 parse_status 设置为 'failed' 并在 parse_error_msg 中存储错误信息

### 需求：启用解析内容的人工审核
系统必须允许人工审核员在 AI 理解之前验证解析质量。

#### 场景：提交解析内容供审核
- **当** 解析成功完成
- **则** 系统将 parse_reviewed 标记为 False 并通知审核员

#### 场景：批准解析内容
- **当** 审核员批准解析内容
- **则** 系统将 parse_reviewed 设置为 True 并记录审核员和时间戳

#### 场景：触发重新解析
- **当** 审核员拒绝解析内容并请求重新解析
- **则** 系统将 parse_status 重置为 'pending' 并触发新的解析任务
