## 新增需求

### 需求：提供结构化理解 API
系统必须提供 API 端点供下游 AI 代理获取已批准的需求理解。

#### 场景：获取已批准的理解
- **当** 下游代理请求已批准需求的理解
- **则** 系统返回包含功能、变更、验收标准和质量问题的结构化 JSON

#### 场景：在响应中包含模块上下文
- **当** 下游代理请求理解
- **则** 系统在响应中包含积累的模块知识

#### 场景：在响应中包含相关需求
- **当** 下游代理请求理解
- **则** 系统在响应中包含相关需求数据

#### 场景：拒绝未批准理解的请求
- **当** 下游代理请求未批准需求的理解
- **则** 系统返回状态码 403 的错误

### 需求：支持批量理解检索
系统必须允许下游代理在一次请求中获取多个需求理解。

#### 场景：按 ID 获取多个理解
- **当** 下游代理提供需求 ID 列表
- **则** 系统返回已批准需求的理解对象数组

#### 场景：在批量请求中过滤未批准的需求
- **当** 批量请求包含未批准的需求
- **则** 系统仅返回已批准的理解并列出跳过的 ID

### 需求：提供理解元数据
系统必须包含关于理解版本和审核状态的元数据。

#### 场景：包含版本信息
- **当** 返回理解数据
- **则** 系统包含版本号和最后更新时间戳

#### 场景：包含审核信息
- **当** 返回理解数据
- **则** 系统包含审核员用户名和审核时间戳

### 需求：支持理解变更通知
系统必须在理解更新时通知下游代理。

#### 场景：理解批准时通知
- **当** 理解被批准
- **则** 系统向注册的下游代理触发 webhook 或通知

#### 场景：理解重新生成时通知
- **当** 理解被重新生成并重新批准
- **则** 系统通知下游代理版本变更

### 需求：提供理解搜索和过滤
系统必须允许下游代理按理解属性搜索和过滤需求。

#### 场景：按重要性级别过滤
- **当** 下游代理按 importance='核心' 过滤
- **则** 系统仅返回包含核心功能的需求

#### 场景：按 MoSCoW 优先级过滤
- **当** 下游代理按 moscow='Must' 过滤
- **则** 系统仅返回包含 Must-have 功能的需求

#### 场景：按功能关键词搜索
- **当** 下游代理搜索功能关键词
- **则** 系统返回包含匹配功能的需求

### 需求：提供质量问题过滤
系统必须允许下游代理按质量问题类型过滤需求。

#### 场景：过滤有质量问题的需求
- **当** 下游代理请求有质量问题的需求
- **则** 系统返回 ai_quality_issues 不为空的需求

#### 场景：按特定问题类型过滤
- **当** 下游代理按 issue_type='风险' 过滤
- **则** 系统返回包含风险类型质量问题的需求

### 需求：支持下游代理认证
系统必须在提供理解数据前对下游 AI 代理进行认证。

#### 场景：验证代理 API 密钥
- **当** 下游代理使用 API 密钥发出请求
- **则** 系统在返回数据前验证密钥

#### 场景：拒绝未认证的请求
- **当** 下游代理在没有有效凭据的情况下发出请求
- **则** 系统返回 401 未授权错误

### 需求：提供理解消费指标
系统必须跟踪哪些下游代理消费理解数据。

#### 场景：记录消费事件
- **当** 下游代理获取理解
- **则** 系统记录代理标识符、需求 ID 和时间戳

#### 场景：提供消费分析
- **当** 管理员请求消费指标
- **则** 系统返回关于哪些代理消费哪些需求的统计信息

### 需求：支持理解格式自定义
系统必须允许下游代理请求特定的数据格式。

#### 场景：请求最小格式
- **当** 下游代理请求 format='minimal'
- **则** 系统仅返回功能和验收标准

#### 场景：请求完整格式
- **当** 下游代理请求 format='full'
- **则** 系统返回包含所有上下文和元数据的完整理解

### 需求：提供理解差异 API
系统必须允许下游代理获取理解版本之间的变更。

#### 场景：获取理解差异
- **当** 下游代理请求版本之间的差异
- **则** 系统返回显示新增、修改和删除功能的结构化差异

#### 场景：处理版本未找到
- **当** 请求的版本不存在
- **则** 系统返回 404 错误及可用版本
