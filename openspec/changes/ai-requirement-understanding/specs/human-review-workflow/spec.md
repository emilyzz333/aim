## 新增需求

### 需求：跟踪解析审核状态
系统必须跟踪解析内容是否已被人工审核。

#### 场景：解析后初始化审核状态
- **当** 文档解析完成
- **则** 系统将 parse_reviewed 设置为 False

#### 场景：记录审核员信息
- **当** 审核员批准解析内容
- **则** 系统记录 parse_reviewed_by 用户和 parse_reviewed_at 时间戳

### 需求：跟踪 AI 理解审核状态
系统必须跟踪 AI 理解是否已被人工审核和批准。

#### 场景：理解后初始化审核状态
- **当** AI 理解完成
- **则** 系统将 ai_reviewed 设置为 False

#### 场景：记录理解审核员信息
- **当** 审核员批准 AI 理解
- **则** 系统记录 ai_reviewed_by 用户和 ai_reviewed_at 时间戳

### 需求：提供解析审核界面
系统必须提供用于审核解析内容的 API 端点。

#### 场景：获取解析内容供审核
- **当** 审核员请求解析内容
- **则** 系统返回 parsed_content 和 parsed_content_with_images 两个字段

#### 场景：通过 API 批准解析内容
- **当** 审核员提交批准
- **则** 系统更新 parse_reviewed 状态并触发 AI 理解任务

#### 场景：拒绝并请求重新解析
- **当** 审核员拒绝解析内容
- **则** 系统重置 parse_status 并允许使用可选参数重新解析

### 需求：提供 AI 理解审核界面
系统必须提供用于审核 AI 理解结果的 API 端点。

#### 场景：获取理解供审核
- **当** 审核员请求 AI 理解
- **则** 系统返回包含功能、质量问题和元数据的结构化理解

#### 场景：通过 API 批准理解
- **当** 审核员提交批准
- **则** 系统更新 ai_reviewed 状态并使理解可供下游消费

#### 场景：拒绝并请求重新生成
- **当** 审核员拒绝理解
- **则** 系统重置 ai_status 并允许使用可选反馈重新生成

### 需求：通知审核员待审核内容
系统必须在内容准备好审核时通知相应的审核员。

#### 场景：解析完成后通知
- **当** 解析成功完成
- **则** 系统向指定审核员发送通知

#### 场景：理解完成后通知
- **当** AI 理解成功完成
- **则** 系统向指定审核员发送通知

### 需求：在下游消费前强制审核
系统必须防止未批准的理解被下游代理消费。

#### 场景：阻止下游访问未批准的理解
- **当** 下游代理请求 ai_reviewed=False 的理解
- **则** 系统返回错误，指示需要批准

#### 场景：允许下游访问已批准的理解
- **当** 下游代理请求 ai_reviewed=True 的理解
- **则** 系统返回完整的理解数据

### 需求：支持审核工作流权限
系统必须对审核操作强制执行基于角色的权限。

#### 场景：限制审核给授权角色
- **当** 没有审核权限的用户尝试批准
- **则** 系统返回权限拒绝错误

#### 场景：允许管理员审核
- **当** 管理员用户提交审核
- **则** 系统接受审核操作

### 需求：跟踪审核历史
系统必须维护审核操作的审计跟踪。

#### 场景：记录批准操作
- **当** 审核员批准内容
- **则** 系统记录批准操作及时间戳和用户

#### 场景：记录拒绝操作
- **当** 审核员拒绝内容
- **则** 系统记录拒绝操作及原因和时间戳

### 需求：在 UI 中显示审核状态
系统必须在用户界面中显示审核状态和待审核项。

#### 场景：显示解析审核状态
- **当** 用户查看需求详情
- **则** 系统显示 parse_reviewed 状态和审核员信息

#### 场景：显示理解审核状态
- **当** 用户查看需求详情
- **则** 系统显示 ai_reviewed 状态和审核员信息

#### 场景：突出显示待审核项
- **当** 用户查看需求列表
- **则** 系统突出显示有待审核的需求
